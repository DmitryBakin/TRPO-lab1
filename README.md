1. Постановка задачи

Разработать программу с консольным интерфейсом на Qt, которая выполняет слежение за выбранными пользователем файлами.

Ограничиться двумя характеристиками файлов, за изменениями которых выполняется слежение:

1. Существование файла — существует файл в файловой системе или нет.
2. Размер файла — текущий размер.

Существует несколько ситуаций для наблюдаемого файла:

1. Файл существует , файл не  пустой - на экран выводится факт существования файла и его размер.
2. Файл существует, файл был изменен - на экран выводится факт существования файла, сообщение о том что файл был изменен и его размер.  
3. Файл не существует - на экран выводится информация о том что файл не существует.

- При возникновении изменения состояния наблюдаемого файла программа должна выводить на консоль соответствующее сообщение.
- В реализации обязательно использовать механизм сигнально-слотового соединения Qt для обеспечения обработки события изменения наблюдаемого файла.
- Программа не должна завершаться после первого изменения — мониторинг ведётся непрерывно до принудительного завершения.

2. Предлагаемое решение

В основе решения лежит активное наблюдение с фиксированным интервалом времени. Каждые 100 мс программа проверяет состояние каждого отслеживаемого файла по двум характеристикам:

-существует ли файл;
-каков его размер.

При обнаружении различий между текущим и предыдущим состоянием генерируются сигналы, которые через сигнально-слотовые соединения передаются в объект вывода (стратегию), где формируется и выводится соответствующее сообщение.

Структура решения (сущности и классы):
FileManager - Синглтон, управляющий списком файлов, таймером и состояниями. Генерирует сигналы при изменениях.
IFileLogs - Логгер. Содержит слот print(const QFileInfo&).
PrintInfoToConsole - выводит информацию в консоль через qDebug(), наследуется от IFileLogs.
PrintInfoToOtherFile - выводит информацию в файл output.txt, наследуется от IFileLogs.

UML-Диаграмма: https://drive.google.com/file/d/1v2J8wSgrU51x2oSyV8hTaUgf0oy2tCNs/view?usp=drive_link

Применённые паттерны:
Паттерн Singleton (Одиночка)
Назначение: гарантирует, что класс имеет только один экземпляр.

3. Инструкция пользователя

Создайте файл input.txt.

Структура input.txt:
-Пути к файлам разделяются запятыми.
-Каждый путь должен быть полным.

Добавьте пути файлов, которые будут наблюдаться.
Запустите программу.

Тест 1) Создание файла

Запуск программы без файла -> создать файл

Тест 2) Изменение файла

Файл существует → изменить содержимое

Тест 3) Удаление файла

Файл существует → удалить
Вывод:

Тест 4) Многократное изменение

Изменить файл 3 раза подряд

"24.04.2026 11:26:57"  Path: "C:/TRPO-lab1/FileMonitoring/text.txt"  Exists: true  Size: 14

Тест 5) Вставка файла с ненулнвым размером

Вывод:
"24.04.2026 11:34:01"  Path: "C:/TRPO-lab1/FileMonitoring/text.txt"  Exists: true  Size: 8

Тест 6) Дубликаты в input.txt

Путь указан дважды. Файл добавляется 1 раз (contains проверка)

Тест 7) Смена Логера

Вызов setOutputMethod(new PrintInfoToOtherFile());
Вывод переключается в output.txt.

Тест 8) nullptr в setOutputMethod

Вызов instance.setOutputMethod(nullptr);
Вывод будет работать с тем же логером, что и до этого.
